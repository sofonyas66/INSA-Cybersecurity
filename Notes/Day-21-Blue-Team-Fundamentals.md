## What is Blue Teaming?

The Blue Team is the defensive side of cybersecurity. Their job is to **detect, respond to, and prevent** attacks against an organisation's systems. While Red Teams simulate attackers, Blue Teams operate and maintain the defences.

Core responsibilities:
- **Continuous monitoring** — 24/7 observation of network traffic, endpoints, and logs via SIEM
- **Threat detection** — identifying indicators of compromise (IoCs) and anomalous behaviour
- **Incident response** — executing playbooks when a security event is confirmed
- **Threat hunting** — proactively searching for threats that evade automated detection
- **Security hardening** — configuring firewalls, EDR, IDS/IPS, and access controls
- **Vulnerability management** — regular scanning, patching, and remediation

| Team | Role | Mindset |
|------|------|---------|
| Blue Team | Defend & Detect | Defender |
| Red Team | Attack & Exploit | Attacker |
| Purple Team | Validate & Improve | Collaborator |

The **MITRE ATT&CK framework** is the Blue Team's primary reference — it maps adversary tactics, techniques, and procedures (TTPs) so defenders can build detection rules aligned to real-world attacks.

## Types of Logs

Logs are timestamped records of events generated by operating systems, applications, network devices, and security tools. Without comprehensive logging, a Blue Team is essentially blind.

| Log Type | Source | Key Data |
|----------|--------|----------|
| **System logs** | OS kernel, services | Login events, process creation, crashes. Windows Event IDs: 4624 (logon), 4625 (failed logon), 4688 (process creation) |
| **Network logs** | Firewalls, IDS/IPS, routers, NetFlow | Src/dst IP, port, protocol, bytes, allow/deny action |
| **Application logs** | Web servers, databases, APIs | HTTP method, URI, status code, user agent, error messages |
| **Authentication logs** | Active Directory, VPN, SSO | Brute-force patterns, account lockouts, impossible travel |
| **DNS logs** | DNS resolvers, Pi-hole, Zeek | C2 beaconing via DGA, DNS tunnelling, high-frequency lookups |
| **Security logs** | EDR, AV, WAF, SIEM | Threat scores, MITRE tags, file hashes, remediation actions |
| **Cloud logs** | AWS CloudTrail, Azure Activity Log | IAM changes, bucket ACL modifications, root account usage |
| **Email logs** | Exchange, O365, Postfix | Sender reputation, attachment hashes, phishing indicators |

## 5W1H Framework

The 5W1H model is the investigative framework for analysing any security event. Every incident must be fully characterised by answering all six questions.

**WHAT** — What happened?
- Type of event (malware, data breach, unauthorised access, policy violation)
- Affected asset and specific action taken (file write, network connection, registry modification)
- Artefacts: file hashes, process names, registry keys, URLs

**WHEN** — When did it happen?
- Exact timestamp in UTC (always normalise to UTC)
- Duration, sequence of related events, recurrence pattern

**WHERE** — Where did it happen?
- Source IP, hostname, geographic region
- Target system or service, network segment (internal vs. external)

**WHO** — Who is responsible?
- Username, UID, service account involved
- Parent/child process chain, machine identity
- Threat actor attribution if possible

**HOW** — How was it done?
- Attack vector: phishing, exploit, brute force, insider, supply chain
- MITRE ATT&CK technique ID mapped to the observed behaviour
- Tools identified, evasion methods used (obfuscation, LOLBins, process injection)

**WHY** — Why did it happen?
- Adversary motivation: financial, espionage, hacktivism, disruption
- Root cause: unpatched vulnerability, misconfiguration, weak credentials, no MFA
- Business impact: data breach, ransomware, service outage, regulatory penalty

## SOC Analysis

A **Security Operations Centre (SOC)** is a centralised team responsible for monitoring, detecting, analysing, and responding to security incidents in real time.

### Analyst Tiers

| Tier | Role | Responsibilities |
|------|------|-----------------|
| **L1** | Alert Analyst / Triage | Monitor dashboards, triage alerts, escalate confirmed incidents, close false positives |
| **L2** | Incident Responder | Deep-dive investigation, threat correlation, malware triage, remediation |
| **L3** | Threat Hunter / SME | Proactive hunting, custom detection rules, reverse engineering |
| **SOC Manager** | Operations Lead | Team management, metrics, escalation authority |

### Core SOC Process

1. **Alert Triage** — classify each alert as True Positive (TP), False Positive (FP), or Benign True Positive (BTP)
2. **Investigation** — correlate log sources, reconstruct the attack timeline, determine scope
3. **Containment** — isolate affected hosts, block malicious IPs/domains, revoke credentials
4. **Eradication** — remove malware, close persistence mechanisms, patch the exploited vulnerability
5. **Recovery** — restore from clean backups, validate integrity, re-enable access with monitoring
6. **Post-Incident Review** — lessons learned, update detection rules, refine playbooks

### Key SOC Metrics

- **MTTD** (Mean Time to Detect) — target: < 1 hour
- **MTTR** (Mean Time to Respond) — target: < 4 hours
- **False Positive Rate** — high FPR leads to analyst fatigue
- **Escalation Rate** — percentage of L1 alerts escalated to L2

## Incident Response (IR)

IR is the structured methodology for managing the full lifecycle of a security incident. The primary framework is **NIST SP 800-61**.

### NIST IR Lifecycle

1. **Preparation** — IR plans, playbooks, tooling, training, tabletop exercises
2. **Detection & Analysis** — validate alerts, apply 5W1H, assign severity, preserve evidence
3. **Containment** — short-term isolation + long-term network-wide controls
4. **Eradication** — remove root cause, purge persistence, patch vulnerability
5. **Recovery** — restore from clean state, validate, monitor before closing
6. **Post-Incident Activity** — lessons learned, IR report, update rules, share IoCs with CTI

### Severity Classification

| Level | Criteria | SLA |
|-------|----------|-----|
| P1 Critical | Active breach, ransomware, data exfiltration | Immediate — 15 min |
| P2 High | Confirmed compromise, lateral movement | < 1 hour |
| P3 Medium | Suspicious activity, unverified threat | < 4 hours |
| P4 Low | Informational alert, minor policy breach | < 24 hours |

## Cyber Threat Intelligence (CTI)

CTI is the process of collecting, processing, analysing, and disseminating information about current and potential threats — transforming raw data into **actionable intelligence**.

### Intelligence Lifecycle

1. **Direction** — define Priority Intelligence Requirements (PIRs)
2. **Collection** — OSINT, dark web, threat feeds, honeypots, ISAC sharing
3. **Processing** — normalise and structure raw data (STIX/JSON format)
4. **Analysis** — identify patterns, attribute actors, assess likelihood and impact
5. **Dissemination** — deliver finished intelligence to the right stakeholders
6. **Feedback** — refine future collection based on stakeholder evaluation

### CTI Levels

| Level | Audience | Content |
|-------|----------|---------|
| **Strategic** | Executives / Board | Threat actor motivations, geopolitical context, industry targeting |
| **Operational** | SOC Managers, IR Leads | Specific campaigns, adversary TTPs, planned attacks |
| **Tactical** | SIEM / EDR / Firewall | IoCs — IPs, domains, file hashes, YARA rules, Sigma detections |

### Key Standards

- **STIX** — JSON-based language for describing CTI objects (IoCs, TTPs, campaigns)
- **TAXII** — protocol for sharing STIX data over HTTPS
- **MISP** — open-source threat intelligence sharing platform
- **YARA** — pattern-matching rules for malware identification
- **Sigma** — generic SIEM rule format, portable across platforms

## How to Analyse a Log

A repeatable 7-step methodology applicable to any log source:

1. **Collect & Centralise** — aggregate into SIEM, normalise timestamps to UTC, verify log completeness
2. **Parse & Normalise** — extract structured fields (timestamp, src IP, dst IP, user, action, result) using a common schema (ECS, CIM)
3. **Establish Baseline** — understand what normal looks like: login hours, traffic volumes, process parent-child chains
4. **Apply Detection Logic** — run correlation rules, threshold alerts, Sigma rules, MITRE-aligned detections
5. **Contextualise & Correlate** — never analyse a single event in isolation; build a timeline across sources
6. **Answer the 5W1H** — document all six questions; every unanswered question is a gap in the investigation
7. **Determine Disposition** — classify as TP, FP, or BTP; escalate, tune, or close accordingly

### Useful Linux Commands

```bash
# Count failed SSH logins per IP
grep 'Failed password' /var/log/auth.log | awk '{print $11}' | sort | uniq -c | sort -rn

# Check for successful logins from suspicious IPs
grep 'Accepted' /var/log/auth.log

# Watch live authentication events
tail -f /var/log/auth.log | grep 'Failed\|Accepted'

# Find cron modifications (persistence indicator)
grep -i 'cron' /var/log/syslog | grep 'EDIT\|new'

# Check systemd errors in the last 24 hours
journalctl -p err -xe --since '24 hours ago'

# Find large outbound transfers in nginx logs
awk '{print $7, $10}' /var/log/nginx/access.log | sort -k2 -rn | head -20
```

## Antivirus and EDR

### Antivirus (AV)

Traditional antivirus detects known malware using **signature-based detection** — comparing file hashes and byte patterns against a database of known threats. It is reactive: it can only catch what it has already seen.

Limitations:
- Cannot detect zero-days or novel malware
- Easily bypassed by obfuscation, packing, or polymorphic code
- No visibility into post-execution behaviour

### EDR (Endpoint Detection and Response)

EDR is the modern evolution of AV. It provides **continuous monitoring and recording of endpoint activity** — processes, file writes, network connections, registry changes — and applies behavioural analytics and threat intelligence to detect both known and unknown threats.

| Feature | Antivirus | EDR |
|---------|-----------|-----|
| Detection method | Signature-based | Signature + Behaviour + ML |
| Scope | File scanning | Full endpoint telemetry |
| Response capability | Quarantine file | Isolate host, kill process, rollback |
| Threat hunting | No | Yes |
| Forensic data | No | Yes — process trees, timelines |
| Zero-day detection | Limited | Yes (behavioural) |

Common EDR platforms: CrowdStrike Falcon, Microsoft Defender for Endpoint, SentinelOne, Carbon Black.

### EDR Key Capabilities

- **Process tree visibility** — see exactly what spawned what (parent → child process chains)
- **Memory scanning** — detect fileless malware living only in RAM
- **Network telemetry** — record all outbound connections from each endpoint
- **Automated response** — isolate a compromised host with one click or automatically
- **Threat hunting** — query across all endpoints for IoCs historically

## Summary — Blue Team Core Principles

1. **Assume breach** — operate as if adversaries are already inside; hunt continuously
2. **Log everything** — gaps in logging are blind spots; you cannot detect what you cannot see
3. **Correlate, never isolate** — single events are rarely meaningful; patterns are
4. **Know your baseline** — you cannot spot anomalies without knowing what normal looks like
5. **Threat-intel-driven defence** — prioritise detections based on who targets you and how
6. **Document rigorously** — if it isn't written down, it didn't happen
7. **Continuously improve** — every incident is an opportunity to build a better defence
